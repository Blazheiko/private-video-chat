# Технічне завдання: Приватний відеочат

**Версія:** 1.6
**Дата:** 2026-07-14
**Кодова назва:** private-video-chat

---

## 1. Загальний опис

Веб-застосунок для приватного спілкування без збереження даних. Користувач відкриває сторінку, створює унікальне посилання (лінку) одного з двох типів і ділиться ним зі співрозмовниками:

- **Групова лінка** — багатокористувацький текстовий чат. Дозволяє **тільки обмін повідомленнями** (через сервер, з наскрізним E2E-шифруванням).
- **Приватна лінка (peer-to-peer)** — спілкування один-на-один. Дозволяє **повідомлення, передачу файлів, аудіодзвінки та відеодзвінки** повністю через WebRTC (P2P), без проходження медіа/даних через сервер.

### 1.1 Ключовий принцип приватності

> **Жодні дані про переписку, файли чи медіапотоки ніде не зберігаються.**

- Сервер не має бази даних для повідомлень, файлів чи метаданих переписки.
- Сервер виступає **тільки** як:
  1. роздавач статики (SPA-фронтенд);
  2. WebSocket-сигнальний сервер для встановлення WebRTC-з'єднань;
  3. реле-транспорт для повідомлень **групового** чату (без персистенції; повідомлення передаються в зашифрованому вигляді й одразу забуваються).
- Кімнати існують **тільки в оперативній пам'яті** процесу, поки активна хоча б одна WebSocket-сесія. Після виходу всіх учасників кімната знищується після короткого **grace-періоду** (щоб пережити реконект/ICE-restart), без будь-якої персистенції вмісту (**ефемерна кімната**, див. §5.4).
- Приватний P2P-чат: сервер бачить лише сигнальні повідомлення (SDP/ICE) для встановлення з'єднання; після з'єднання весь трафік іде напряму між учасниками.

## 2. Цілі та нецілі

### 2.1 Цілі
- Миттєве створення приватної кімнати без реєстрації та автентифікації.
- Мінімальний слід даних: нічого не зберігається на сервері поза життям активної сесії.
- Надійне встановлення WebRTC-з'єднань, зокрема за симетричним NAT (через власний TURN).
- Легкий, швидкий фронтенд на сигналах.

### 2.2 Нецілі (поза межами v1.0)
- Реєстрація/акаунти користувачів, історія чатів.
- Групові аудіо/відеодзвінки (SFU/MCU). У v1.0 дзвінки — тільки 1-на-1 у приватній лінці.
- Мобільні нативні застосунки (тільки веб; PWA — опційно на майбутнє).
- Модерація, ролі, адміністрування кімнат.

## 3. Технологічний стек

| Шар | Технологія |
|-----|-----------|
| Монорепозиторій | **pnpm workspaces** |
| Мова | **TypeScript** (strict) |
| Бекенд (WebSocket / HTTP) | **uWebSockets.js**, без веб-фреймворків |
| Реле/сигналінг | Власний протокол поверх WebSocket |
| P2P-медіа та дані | **WebRTC** (RTCPeerConnection, DataChannel) |
| NAT-обхід | **STUN + власний TURN (coturn)** |
| Фронтенд-реактивність | **Preact + @preact/signals** |
| Збірка фронтенду | **Vite** |
| Пакетний менеджер | pnpm |
| Рантайм бекенду | Node.js (LTS, ≥ 20) |

### 3.1 Обґрунтування вибору
- **uWebSockets.js** — найпродуктивніша реалізація WebSocket для Node.js, мінімальний overhead, вбудований HTTP-сервер для роздачі статики. Фреймворк не потрібен, бо API проєкту вузьке (роздача SPA + WS-сигналінг).
- **Preact + @preact/signals** — ~4 KB рантайм, зріла реактивність на сигналах, JSX, велика екосистема. Найбезпечніший вибір серед «мінімалістичних на сигналах».
- **coturn** — стандарт де-факто для TURN; гарантує з'єднання навіть за суворих NAT/фаєрволів (~10–20% з'єднань інакше не встановлюються).

## 5. Функціональні вимоги

### 5.1 Стартова сторінка
- FR-1. Користувач відкриває головну сторінку без будь-якої автентифікації.
- FR-2. Доступні дві кнопки створення кімнати:
  - «Створити груповий чат» → генерує **групову лінку**;
  - «Створити приватний чат (P2P)» → генерує **приватну лінку**.
- FR-3. Після створення показується посилання з кнопкою «Скопіювати» та (опційно) QR-код.
- FR-4. Ідентифікатор кімнати — криптографічно випадковий (напр. 128-біт, base64url).
- FR-5. **Ключ E2E-шифрування генерується на клієнті й вбудовується у fragment (`#k=...`) лінки.** Fragment не надсилається на сервер — сервер ніколи не бачить ключ. Після зчитування fragment стирається з URL (§8.1); QR-код (FR-3) генерується виключно на клієнті.

Приклад лінки:
```
https://host/r/g/AbC123...#k=<base64url-key>   ← групова
https://host/r/p/XyZ789...#k=<base64url-key>   ← приватна (P2P)
```

### 5.2 Груповий чат (тип «group»)
- FR-6. Дозволено **тільки** обмін текстовими повідомленнями.
- FR-7. Повідомлення шифрується на клієнті (ключ із fragment лінки) → надсилається на сервер → сервер ретранслює всім іншим учасникам кімнати → розшифровується на їхніх клієнтах.
- FR-8. Сервер **не зберігає** повідомлення; ретрансльоване повідомлення не логується й не персиститься.
- FR-9. Новий учасник, який приєднався, **не отримує** попередніх повідомлень (немає історії).
- FR-10. Кількість учасників групового чату **без застосункового ліміту за задумом**, але з **м'яким операційним cap** `MAX_GROUP_PARTICIPANTS` (дефолт 50, §10.5) для захисту від fanout-амплікації; перевищення → `error{ROOM_FULL}` новому учаснику. Cap конфігурований.
- FR-11. Показ списку учасників (за псевдонімом/аватаром-кольором, що генерується локально), індикатори «набирає повідомлення» — опційно.
- FR-12. Файли, аудіо- та відеодзвінки у груповому чаті **недоступні**.

### 5.3 Приватний чат (тип «private», P2P)
- FR-13. Розрахований на **двох учасників** (1-на-1). Третій, хто відкрив лінку, отримує повідомлення «кімната зайнята».
- FR-14. Після приєднання другого учасника встановлюється WebRTC-з'єднання через сигналінг сервера.
- FR-15. Доступний функціонал повністю через WebRTC (P2P):
  - **Текстові повідомлення** — через RTCDataChannel;
  - **Передача файлів** — через RTCDataChannel (чанкінг, прогрес);
  - **Аудіодзвінок** — getUserMedia(audio) + RTCPeerConnection;
  - **Відеодзвінок** — getUserMedia(video+audio) + RTCPeerConnection.
- FR-16. Медіа та дані **не проходять через сервер** (окрім TURN-relay-кандидатів, якщо прямий P2P неможливий; TURN не розшифровує медіа — DTLS/SRTP).
- FR-17. Керування дзвінком: почати/завершити, вимкнути мікрофон, вимкнути камеру, перемкнути камеру (front/back на мобільних).
- FR-18. Індикація стану з'єднання (connecting / connected / reconnecting / failed).

### 5.4 Життєвий цикл кімнати (ефемерна, відновлювана)

Семантику **«одноразова кімната» прибрано** як хибну (див. §7.3: сама лінка з `#k` багаторазова, тож знищення в пам'яті не робить кімнату криптографічно одноразовою й не дає forward secrecy). Замість неї — **ефемерна кімната з grace-періодом** та явним `roomInstanceId`.

- FR-19. Кімната створюється в пам'яті при першому WS-підключенні за її ID й отримує **`roomInstanceId`** — випадковий ідентифікатор цього інстансу (входить у `joined`; змінюється при кожному новому інстансі того самого room ID).
- FR-20. Після відключення **останнього** учасника кімната не знищується миттєво: запускається **grace-період** (`ROOM_GRACE_MS`, напр. 15–30 с). Так само, коли окремий учасник втратив сокет, його **`participantId`-слот** тримається протягом grace (не миттєвий `peer-left`).
  - Реконект у межах grace з валідним **`resumeToken`** (§6.1) → відновлюється **той самий інстанс** і **той самий `participantId`-слот** (`roomInstanceId` зберігається; `joined{resumed:true}`). Покриває реконект WS (NFR-10) та ICE-restart (NFR-11).
  - Реконект без/з невалідним токеном → `BAD_RESUME_TOKEN`, приєднання як новий учасник (якщо є вільний слот).
  - Якщо grace минув без підключень — інстанс знищується з усім станом; по peer-ах, чий слот протух, надсилається `peer-left{reason:'grace-expired'}`.
- FR-20a. **Tombstone для приватних кімнат.** Після завершення grace приватної кімнати її room ID заноситься в короткоживучий **tombstone** (`ROOM_TOMBSTONE_MS`, напр. кілька хвилин): протягом цього вікна нове підключення за тим самим ID **не** створює кімнату (`error{ROOM_CLOSED}`). Прибирає **перезаняття** ID після виходу (§9.3). Tombstone зберігає **лише ID + час** (жодного вмісту). Для групових кімнат не застосовується.
- FR-21. Спроба зайти за лінкою кімнати, якої немає в пам'яті **й немає в tombstone**, створює **новий інстанс** з тим самим ID і новим `roomInstanceId` (для приватного чату перший, хто зайшов, стає «господарем» очікування). Клієнти відрізняють інстанси за `roomInstanceId` і скидають per-`participantId` seq-стан при його зміні.

## 6. Протокол WebSocket (сигналінг + реле)

Спільні типи оголошуються у `packages/shared/src/protocol.ts` і використовуються обома застосунками.

### 6.1 Ідентифікатори (три різні сутності)

Реконект-семантика вимагає розрізняти **сокет** і **учасника**. Тому три сутності:

- **`connId`** — ідентифікатор **конкретного WebSocket-конекту** (сокета). Сервер призначає його при кожному підключенні; він живе **рівно стільки, скільки живе сокет**. Новий WS (у т.ч. після реконекту) → **новий `connId`**. Це суто **транспортна** деталь: у протокольних повідомленнях `connId` **не фігурує** (щоб уникнути суперечності «стабільний ID сокета, що переживає реконект»). Сервер лише мапить `participantId → поточний connId` для доставки.
- **`participantId`** — **стабільна ідентичність вкладки в межах одного `roomInstanceId`**. Сервер призначає її при першому `join` і **зберігає через реконект** (поки триває grace-період кімнати, §5.4). Саме `participantId`:
  1. використовується у `to`/`from` (маршрутизація сигналінгу та атрибуція relay);
  2. є ключем для `seq`/dedup (§6.4, §6.9).
  Клієнт дізнається свій `participantId` з `joined.selfId`; чужі — з `PeerInfo`.
- **`resumeToken`** — **серверний секрет**, що видається при першому `join` (у `joined`) і **не залишає** пари клієнт↔сервер. Щоб **повернути той самий `participantId`-слот** під час grace, клієнт надсилає `resumeToken` у `join`. Без валідного токена сторонній власник room ID **не може** зайняти чужий слот протягом grace (закриває gap slot-squatting раніше за легітимний reconnect, §5.4/§9.3). Токен ротовний: сервер може видати новий `resumeToken` у кожному `joined`.
- **`senderId`** — app-level **opaque ID вкладки** (локально згенерований base64url-ідентифікатор; може бути seed для локального імені/кольору). Це **не** довірене display-name поле й не має містити людський псевдонім напряму. Входить у AAD. **Недовірений** проти інсайдера (§7.2). Довірена атрибуція — `from: participantId` зі штампа сервера.
- **`msgId`** — унікальний ID повідомлення (UUID/random), дедуплікація.
- **`seq`** — монотонний лічильник відправника (anti-replay/order). Прив'язаний до **`participantId`**; **переживає реконект** (не скидається — інакше отримувачі відкинуть повідомлення, §6.4). `cid` (лічильник для ACK, §6.9) — **навпаки, локальний для сокета** й скидається на кожному новому WS.
- **`ts`** — час відправлення (мс). **Не контроль безпеки** (немає довіреного часу): лише UI + широке (±5 хв) допоміжне вікно; replay покриває `seq`.

> **Модель довіри до `participantId`:** сервер довірений **лише щодо чесної маршрутизації та штампа `from`** (він і так бачить, хто на якому сокеті). Це дає практичну атрибуцію відправника проти *інсайдерів*. Проти *зловмисного сервера* (може збрехати про `from`) — лише криптопідписи (§7.2 B). Щодо **вмісту** сервер лишається недовіреним (E2E).

### 6.2 Загальний конверт
```ts
const PROTOCOL_VERSION = 1;

type ClientMessage =
  // participantId клієнт НЕ надсилає — його призначає сервер. resumeToken (за наявності)
  // повертає той самий participantId-слот під час grace (§5.4, §6.1).
  // sessionNonce — 128-біт base64url, transcript §7.1 (тільки private).
  | { v: number; t: 'join'; room: string; kind: 'group' | 'private'; senderId: string; sessionNonce: string; resumeToken?: string; senderPubKey?: string }
  | { v: number; t: 'leave' }
  | { v: number; t: 'relay'; cid: number; env: EncryptedEnvelope }  // cid — WS-локальний лічильник для ACK (§6.9)
  | { v: number; t: 'signal'; to: string; data: RTCSignal }  // to = participantId peer-а
  | { v: number; t: 'auth'; to: string; mac: string }        // HMAC над транскриптом (§7.1)
  | { v: number; t: 'ping' }                                 // heartbeat (§6.11)
  ;

type ServerMessage =
  // selfId/from/peers.participantId — сервер-призначені стабільні ідентичності (§6.1).
  // resumeToken — секрет для reclaim слота під час grace (§6.1); ротовний.
  | { v: number; t: 'joined'; selfId: string; roomInstanceId: string; resumeToken: string; resumed: boolean; peers: PeerInfo[]; ice?: IceConfig }
  | { v: number; t: 'peer-joined'; peer: PeerInfo }
  | { v: number; t: 'peer-left'; participantId: string; reason: 'left' | 'grace-expired' } // reason розрізняє тимчасовий обрив (нема події) і остаточний вихід
  | { v: number; t: 'relay'; from: string; env: EncryptedEnvelope } // from = participantId, штампує сервер
  | { v: number; t: 'signal'; from: string; data: RTCSignal }
  | { v: number; t: 'auth'; from: string; mac: string }
  | { v: number; t: 'ack'; cid: number }                     // підтвердження прийому relay сервером (§6.9)
  | { v: number; t: 'ice-refresh'; ice: IceConfig }          // оновлені TURN-креденшели до TTL (§6.10)
  | { v: number; t: 'pong' }                                 // відповідь на ping (§6.11)
  | { v: number; t: 'error'; code: ErrorCode; message: string; fatal: boolean } // ЄДИНИЙ механізм помилок (§6.8); room-full/room-closed = коди ROOM_FULL/ROOM_CLOSED
  ;

type PeerInfo = {
  participantId: string;      // довірений (сервер-призначений, стабільний у межах roomInstanceId)
  senderId: string;           // app-level псевдонім, оголошений у join (недовірений)
  sessionNonce: string;       // для transcript §7.1 (private)
  senderPubKey?: string;      // зарезервовано під §7.2(B)
};

// Видається сервером при join для private-кімнат і оновлюється до закінчення TTL (§6.10).
// Для group-кімнат поле joined.ice відсутнє: груповий чат не використовує WebRTC/TURN.
type IceConfig = {
  iceServers: RTCIceServer[]; // STUN + TURN з короткочасними креденшелами
  ttlSec: number;             // час життя TURN-креденшелів
  policy: 'all' | 'relay';    // iceTransportPolicy
};

// Шифротекст + автентифіковані метадані.
// Поля msgId/senderId/seq/ts входять в AAD — автентифіковані тегом GCM, але не приховані.
// ГАРАНТІЯ: тег AES-GCM захищає AAD від підміни ТИМ, ХТО НЕ МАЄ КЛЮЧА (насамперед — сервером).
// ОБМЕЖЕННЯ: інсайдер із тим самим спільним ключем МОЖЕ підробити senderId/seq (див. §7.2).
// Довірена атрибуція відправника — за штампом сервера from=participantId, а не за senderId в AAD.
type EncryptedEnvelope = {
  iv: string;            // base64url, унікальний на кожне повідомлення
  ciphertext: string;    // base64url, AES-GCM
  aad: {                 // authenticated associated data (не шифрується, але автентифікується)
    msgId: string;
    senderId: string;
    seq: number;
    ts: number;
  };
  sig?: string;          // опційний Ed25519-підпис над (iv||ciphertext||aad). Зарезервовано під §7.2(B); у v1 ігнорується.
};

type RTCSignal =
  | { type: 'offer' | 'answer'; sdp: string }
  | { type: 'ice'; candidate: RTCIceCandidateInit };

// Закритий стабільний перелік (§6.8). Додавати лише в кінець.
type ErrorCode =
  | 'PROTOCOL_VERSION' | 'BAD_MESSAGE' | 'MSG_TOO_LARGE'
  | 'NOT_JOINED' | 'ALREADY_JOINED' | 'ROOM_FULL' | 'ROOM_CLOSED'
  | 'WRONG_KIND' | 'UNKNOWN_PEER' | 'RATE_LIMITED'
  | 'JOIN_TIMEOUT' | 'BACKPRESSURE' | 'BAD_RESUME_TOKEN'
  | 'CAPACITY'      // вичерпано MAX_ROOMS/MAX_CONNECTIONS — не помилка, а ліміт ємності (§6.8, §10.5)
  | 'INTERNAL';
```

### 6.3 Правила сервера
- Сервер призначає `participantId` при першому `join`, мапить його на поточний `connId` (сокет) і **штампує `from: participantId`** на кожному маршрутизованому/ретрансльованому повідомленні. Клієнтські поля з чужим `from` ігноруються.
- **Реконект:** `join` з валідним `resumeToken` протягом grace (§5.4) → сервер **відновлює той самий `participantId`-слот**, перемапує його на новий `connId`, відповідає `joined{resumed:true}`. Невалідний/протухлий токен → `BAD_RESUME_TOKEN` (клієнт стартує як новий учасник). Це не «новий сокет = новий ID»: стабільний ключ — `participantId`, сокет під ним змінний.
- Сервер маршрутизує `signal` та `auth` лише між двома peer-ами однієї приватної кімнати (за `to = participantId`, доставка на поточний `connId`).
- Сервер ретранслює `relay` усім peer-ам групової кімнати, крім відправника.
- Сервер **не парсить і не логує** `ciphertext` (E2E) та SDP-вміст.
- Сервер **не переставляє** `relay` й ретранслює `env` як є; контроль порядку/replay — на клієнті за `seq`.
- Валідація кожного вхідного повідомлення (рантайм-схема, §6.13): `v` (§6.7), розміри/формати (§6.6), тип кімнати (`WRONG_KIND`), стан (§6.12), rate-limit/fanout (§10.5). Помилки → §6.8.
- `signal`/`auth`: `to` мусить бути **наявним peer-ом тієї самої приватної кімнати**; інакше `UNKNOWN_PEER`.

### 6.4 Клієнтські інваріанти автентичності
- Довірена ідентичність відправника — **`from = participantId`** зі штампа сервера (не `senderId` з AAD). UI прив'язує ім'я/колір до `participantId`.
- Клієнт **відхиляє** `relay` із `seq`, що не більший за останній прийнятий від цього **`participantId`** (anti-replay). `seq`-стан ведеться per-`participantId`.
- **Скидання `lastSeq`/dedup — лише при остаточному виході** учасника (`peer-left{reason:'grace-expired'}`) або зміні `roomInstanceId`. Тимчасовий обрив/реконект peer-а **не** генерує `peer-left` і **не** скидає стан — тому збережений `seq` після реконекту лишається валідним (усуває суперечність «reconnect ⇄ per-`connId` seq»).
- `ts` — лише для UI/широке вікно (§6.1).
- Клієнт дедуплікує за `msgId`.
- AAD перевіряється тегом AES-GCM: підміна поля **сервером** (без ключа) → провал розшифрування. Проти інсайдера з ключем AAD не захищає (§7.2).

### 6.5 Узгодження WebRTC у приватній кімнаті (perfect negotiation)
Щоб уникнути glare (одночасні offer з обох боків), застосовуємо шаблон **polite / impolite peer**:
- Peer, який **уже був** у кімнаті на момент приєднання другого, — **impolite** (ініціює offer, при колізії ігнорує вхідний offer).
- Peer, який **приєднався другим**, — **polite** (при колізії відкочує власний offer і приймає вхідний).
- Роль визначається детерміновано з `joined.peers` (порожній список → impolite-очікувач; непорожній → polite-приєднувач).
- Використовується стандартний потік `onnegotiationneeded` + `RTCPeerConnection.rollback`.

### 6.6 Ліміти розмірів та формати (нормативно)
Усі ліміти конфігуровані (константи в `packages/shared/src/constants.ts`); значення нижче — дефолти. Перевищення → `error` з кодом `MSG_TOO_LARGE` і розрив (`fatal:true`).

| Поле / повідомлення | Ліміт (дефолт) | Формат |
|---------------------|----------------|--------|
| WS-фрейм (будь-яке повідомлення) | 64 KiB | UTF-8 JSON, один фрейм = одне повідомлення |
| `room` (ID кімнати) | 22 символи | base64url від 128 біт (§5.1, FR-4) |
| `participantId` | 22 символи | base64url від 128 біт (сервер-призначений) |
| `resumeToken` | 64 байти ентропії; до 86 символів encoded | opaque unpadded base64url (серверний секрет) |
| `senderId` | до 64 символів | opaque unpadded base64url локальної вкладки; не display-name |
| `sessionNonce` | 22 символи | unpadded base64url від 128 біт |
| `senderPubKey` | 43 символи | unpadded base64url Ed25519 (32 байти) |
| `EncryptedEnvelope.ciphertext` | 16 KiB | base64url |
| `EncryptedEnvelope.iv` | 16 символів | base64url 12 байт |
| `EncryptedEnvelope.sig` | 86 символів | unpadded base64url Ed25519 (64 байти) |
| `RTCSignal.sdp` | 16 KiB | рядок SDP |
| `RTCSignal.candidate` | 1 KiB | `RTCIceCandidateInit` |
| `mac` | 43 символи | unpadded base64url HMAC-SHA256 (32 байти) |
| `msgId` | 36 символів | UUIDv4 |
| `seq` | uint53 | зростаючий цілий |
| `ts` | uint53 | epoch ms |

- **Кодування base64url — unpadded** (без `=`) для всіх протокольних полів, якщо явно не зазначено інше. Довжини в таблиці — encoded-довжини для unpadded base64url; поля з лімітом у байтах мають валідувати і decoded size, і максимальну encoded-довжину.
- Числові поля — цілі в діапазоні `[0, 2^53)`. `v` — ціле. Невідомі поля в об'єкті **ігноруються** (forward-compat), але **зайвий розмір** усе одно рахується до ліміту фрейму.

### 6.7 Версіонування та несумісність
Модель — **асиметрична обробка невідомого `t`** (усуває суперечність «новий `t` сумісний» ⇄ «невідомий `t` = BAD_MESSAGE»):

- **Client→Server команда з невідомим `t` → `BAD_MESSAGE` (fatal).** Сервер **мусить** розуміти кожну команду (вона змінює стан), тому невідома команда неприпустима. **Додавання нової client-команди = інкремент `PROTOCOL_VERSION`.**
- **Server→Client подія з невідомим `t` → клієнт мовчки ігнорує.** Нові server-події (напр. майбутній `typing-relay`) — **сумісна зміна**: старий клієнт їх пропускає без помилки. Валідатор клієнта (§6.13) для невідомого server-`t` виконує **skip**, а не `BAD_MESSAGE`.
- **Версійний handshake:** `join` несе `v = PROTOCOL_VERSION`. Сервер, що не підтримує major `v`, → `error{PROTOCOL_VERSION, fatal}` + close `4400`. Downgrade-переговорювання немає (проти downgrade-атак).
- **Опційне поле** в наявному повідомленні — сумісна зміна (не інкрементує major); невідомі поля ігноруються (§6.6).
- Зведення: **major-інкремент** ⇐ нова client-команда, зміна семантики/типу наявного поля, видалення поля. **Сумісно** ⇐ нова server-подія, нове опційне поле.

### 6.8 Коди помилок (стабільний перелік)
`ErrorCode` — закритий рядковий enum; коди стабільні між версіями (додавати лише в кінець).

| Код | Значення | `fatal` | WS close code |
|-----|----------|---------|---------------|
| `PROTOCOL_VERSION` | несумісна версія протоколу | так | 4400 |
| `BAD_MESSAGE` | не-JSON, невідомий `t`, порушення схеми (§6.13) | так | 4400 |
| `MSG_TOO_LARGE` | перевищено ліміт §6.6 | так | 4400 |
| `NOT_JOINED` | повідомлення до `join`/після `leave` | так | 4409 |
| `ALREADY_JOINED` | повторний `join` на тому ж сокеті | ні | — |
| `ROOM_FULL` | приватна кімната зайнята (2/2) або група ≥ `MAX_GROUP_PARTICIPANTS` | так | 4403 |
| `ROOM_CLOSED` | ID у tombstone (§5.4) | так | 4403 |
| `BAD_RESUME_TOKEN` | невалідний/протухлий `resumeToken` — клієнт стартує як новий учасник | ні | — |
| `WRONG_KIND` | `relay` у private або `signal`/`auth` у group | так | 4400 |
| `UNKNOWN_PEER` | `signal.to`/`auth.to` не існує в кімнаті | ні | — |
| `RATE_LIMITED` | перевищено rate-limit (§10.5) | ні* | 4429 |
| `JOIN_TIMEOUT` | не надіслано `join` за `JOIN_TIMEOUT_MS` (§6.11) | так | 4408 |
| `BACKPRESSURE` | вихідний буфер переповнений (§6.11) | так | 4413 |
| `CAPACITY` | вичерпано `MAX_ROOMS`/`MAX_CONNECTIONS` — ліміт ємності, не збій | так | 1013 |
| `INTERNAL` | справжня внутрішня помилка сервера | так | 1011 |

\* `RATE_LIMITED` стає `fatal` після N повторних порушень (§10.5).

> **Єдиний механізм помилок.** Окремих подій `room-full`/`room-closed` **немає** — усі відмови йдуть через `error{code}` (коди `ROOM_FULL`/`ROOM_CLOSED`). `CAPACITY` (WS close `1013 Try Again Later`) відрізняє тимчасове вичерпання ємності від справжнього `INTERNAL` (`1011`).

### 6.9 Доставка, ACK, дедуплікація, replay
- **Модель доставки:** best-effort. Сервер **не буферизує** й **не переставляє** повідомлення (§6.3). Втрата при обриві — за дизайном (§9.3).
- **ACK (backpressure/flow):** на кожен `relay{cid}` сервер, **прийнявши й розіславши** його, повертає відправнику `ack{cid}`. `cid` — монотонний **WS-локальний** лічильник (uint53), що **скидається на кожному новому сокеті** (це суто транспортне підтвердження прийому, не end-to-end). Відсутність `ack` за `ACK_TIMEOUT_MS` → «не підтверджено». `ack` **не** гарантує доставку peer-ам (сервер недовірений), лише прийом сервером.
- **Дедуплікація (отримувач):** за `env.aad.msgId`; вікно останніх M `msgId` per-`participantId` (LRU, дефолт M=1024).
- **Replay/order (отримувач):** приймається лише `env.aad.seq > lastSeq[participantId]`; інакше — відкинути. `lastSeq` per-`participantId`; **скидається лише при остаточному виході** (`peer-left{reason:'grace-expired'}`) або зміні `roomInstanceId` — **не** при тимчасовому реконекті (§6.4).
- Сигнальні (`signal`/`auth`) — не мають ACK; надійність забезпечує WebRTC (ICE-retransmit) та perfect negotiation.

### 6.10 Видача та ротація TURN-креденшелів
- TURN-креденшели **не постійні**: сервер генерує short-term credentials (coturn REST `username = "<expiry>:<rand>"`, `password = base64(HMAC-SHA1(secret, username))`).
- Видаються у складі `joined.ice` (`IceConfig`) при join **лише для private-кімнат**. Для group-кімнат `ice` відсутній, щоб не видавати TURN-креденшели без WebRTC-потреби.
- За `ttlSec * 0.8` до закінчення сервер надсилає `ice-refresh{ice}` з новими креденшелами **тільки private-учасникам**; клієнт застосовує їх до `RTCPeerConnection.setConfiguration` (для майбутніх ICE-restart).
- Секрет coturn ніколи не потрапляє на клієнт.

### 6.11 Транспортні правила (Origin, timeouts, heartbeat, backpressure)
- **Origin validation:** при WS-upgrade сервер перевіряє заголовок `Origin` проти allow-list (свій домен). Невідповідність → відмова upgrade (HTTP 403). Захист від cross-origin WS-hijack.
- **Join timeout:** якщо клієнт не надіслав `join` протягом `JOIN_TIMEOUT_MS` (дефолт 10 с) після відкриття WS → `error{JOIN_TIMEOUT}` + close `4408`.
- **Heartbeat:** клієнт шле `ping` кожні `HEARTBEAT_MS` (дефолт 25 с); сервер відповідає `pong`. Сервер також має власний ping-таймер; за відсутності будь-якого трафіку/пінгу протягом `IDLE_TIMEOUT_MS` (дефолт 60 с) з'єднання дропається. (uWS має вбудований idle-timeout + автопінг — використати його.)
- **Backpressure:** перед `send` сервер перевіряє `ws.getBufferedAmount()`; якщо > `MAX_BACKPRESSURE_BYTES` (дефолт 1 MiB) → `error{BACKPRESSURE}` + close `4413` (повільний споживач не має роздувати пам'ять). Для fanout групи — те саме per-отримувач; переповнений отримувач дропається, решта не страждає.

### 6.12 Стан-машина з'єднання (клієнт)
```
        connect WS
CLOSED ───────────▶ CONNECTING
                        │ ws.open
                        ▼
                    OPEN(no room) ──send join──▶ JOINING
                        ▲                           │
                        │                           │ recv joined
              recv error(non-fatal)                 ▼
                        │                         JOINED ──recv relay/signal/peer-*──▶ (loop)
                        │                           │
        ws.close/error  │                           │ send leave  |  ws.close
                        ▼                           ▼
                   RECONNECTING ◀──grace(§5.4)── CLOSING ──▶ CLOSED
                        │ backoff, re-open + re-join(room, resumeToken, same senderId/seq)
                        └────────────────────────────────────────────▶ JOINING
```
- Дозволені переходи лише за схемою; повідомлення не за станом (напр. `relay` до `JOINED`) → `NOT_JOINED`.
- **Реконект:** `join{room, resumeToken, senderId}` зі збереженим `seq`; `cid` скидається (WS-локальний). Очікувані відповіді:
  - `joined{resumed:true}` з **тим самим** `participantId`/`roomInstanceId` → продовжуємо, стан `seq`/dedup валідний.
  - `joined{resumed:false}` або змінений `roomInstanceId` → **новий** учасник: скинути per-peer `lastSeq`/dedup (§6.9), показати «нова сесія».
  - `error{BAD_RESUME_TOKEN}` → повторний `join` **без** токена як новий учасник.
- `RECONNECTING`: експоненційний backoff з jitter (напр. 0.5→8 с, cap), припиняється після `MAX_RECONNECT_ATTEMPTS` або `error{ROOM_CLOSED|ROOM_FULL|CAPACITY}`.

### 6.13 Рантайм-валідація
- TypeScript-типи **не** валідують мережеві дані. Кожне вхідне повідомлення (на сервері **і** на клієнті) проходить **рантайм-валідацію схеми** перед обробкою.
- Джерело істини — **JSON Schema** (або `zod`/`valibot`-схеми, з яких генерується і TS-тип, і JSON Schema) у `packages/shared`. Схема покриває: набір `t`, обов'язкові/опційні поля, типи, діапазони чисел, довжини рядків, `format: base64url`, ліміти §6.6.
- Провал валідації → `BAD_MESSAGE` (`fatal`), без часткової обробки. Валідатор ігнорує невідомі **поля** (§6.6).
- **Обробка невідомого `t` — асиметрична (§6.7):** валідатор **сервера** на невідомій client-команді → `BAD_MESSAGE`; валідатор **клієнта** на невідомій server-події → **skip** (мовчки ігнорує, не помилка). Це і робить нові server-події сумісною зміною.

### 6.14 DataChannel-протокол (приватна кімната)
Після встановлення P2P відкриваються **іменовані** DataChannel-и (усі поверх DTLS):

| Канал | `label` | `ordered` | `maxRetransmits` | Призначення |
|-------|---------|-----------|------------------|-------------|
| Контроль/текст | `chat` | true | null (reliable) | текстові повідомлення + керуючі кадри файлів |
| Дані файлу | `file:<transferId>` | true | null (reliable) | бінарні чанки одного трансферу |

**Кадри каналу `chat`** — JSON (UTF-8), `{ t, ... }`:
```ts
type ChatFrame =
  | { t: 'msg'; msgId: string; ts: number; text: string }        // текст (≤ 8 KiB)
  | { t: 'msg-ack'; msgId: string }                              // прочитано/доставлено
  | { t: 'typing'; on: boolean }
  | { t: 'file-offer'; transferId: string; name: string; size: number; mime: string; chunkSize: number; sha256?: string } // sha256 — опційний precomputed hash (§6.14.1)
  | { t: 'file-accept'; transferId: string; offset: number }     // offset > 0 = відновлення
  | { t: 'file-reject'; transferId: string; reason?: string }
  | { t: 'file-progress'; transferId: string; received: number } // отримувач → відправнику
  | { t: 'file-complete'; transferId: string; sha256: string }   // фінальний hash (перевірка цілісності)
  | { t: 'file-cancel'; transferId: string }
  | { t: 'file-error'; transferId: string; code: 'CHECKSUM' | 'STALL' | 'TOO_LARGE' | 'IO' } // фатальна помилка трансферу
  ;
```

**Передача файлу (канал `file:<transferId>`):**
1. Відправник шле `file-offer` (`chat`). Отримувач → `file-accept{offset}` або `file-reject`.
2. Відправник відкриває `file:<transferId>`, шле **бінарні** чанки по `chunkSize` починаючи з `offset`. Framing чанка — див. §6.14.2.
3. **Flow-control:** відправник призупиняє надсилання, коли `dc.bufferedAmount > BUFFER_HIGH` (дефолт 1 MiB), відновлює на `bufferedamountlow` (`bufferedAmountLowThreshold = 256 KiB`).
4. Отримувач періодично шле `file-progress` (`chat`). По завершенні відправник шле `file-complete{sha256}`.
5. `file-cancel` з будь-якого боку → закрити `file:<transferId>`, відкинути частковий файл.

#### 6.14.1 Цілісність (checksum) і поведінка при пошкодженні
- **Hash — наприкінці.** SHA-256 обчислюється відправником **інкрементально** у міру читання (див. §6.14.2 про механізм) й надсилається у `file-complete`. Обчислити наперед (в `file-offer`) при потоковому читанні не можна; поле `file-offer.sha256` — **опційне** (лише якщо файл уже прохешований).
- Отримувач рахує SHA-256 інкрементально паралельно й звіряє з `file-complete.sha256`.
- **Умова запуску звірки (міжканальний порядок, §6.14.2):** checksum перевіряється **лише** коли одночасно `receivedBytes === file-offer.size` **і** отримано `file-complete` **і** індекси чанків суцільні від `offset`. Інакше — чекати (не фейлити передчасно).
- **Розбіжність → фатальний фейл усього трансферу** (`file-error{CHECKSUM}`): частковий файл відкидається. У v1 **per-chunk відновлення відсутнє** — новий `file-offer` (окремий `transferId`).

#### 6.14.2 Framing чанків, порядок каналів, offset↔index
- **Framing:** бінарний кадр чанка = `[uint32 BE index][payload bytes]` (4-байтовий заголовок + дані). `index` — 0-based порядковий номер чанка.
- **offset рахується в байтах payload** (без 4-байтових заголовків). Інваріанти:
  - `offset % chunkSize === 0` завжди, окрім значення `offset === size` (файл уже повний → одразу перевірка завершення, канал `file:` не відкривається);
  - `expectedIndex = offset / chunkSize`; перший чанк після `file-accept{offset}` мусить мати саме цей `index`;
  - усі чанки, крім останнього, мають довжину payload рівно `chunkSize`; останній — `size mod chunkSize` (або `chunkSize`, якщо кратно).
- **Обробка аномалій чанків** (отримувач):
  - `index < expectedIndex` (дубль/уже записаний) → **ігнорувати**;
  - `index > expectedIndex` (розрив/gap) → трансфер **фейл** `file-error{IO}` (канал `ordered:true`, тож gap = аномалія, не переставлення);
  - `index`, що виводить сумарний розмір за `file-offer.size` (out-of-range/overflow) → **фейл** `file-error{TOO_LARGE}`;
  - кумулятивно контролювати `receivedBytes ≤ size`.
- **Міжканальний порядок.** `ordered:true` гарантує порядок **у межах одного** каналу, але **не** між `file:<id>` і `chat`. Тому `file-complete` (канал `chat`) може прийти **раніше** за останній бінарний чанк (канал `file:`). Правило вище (§6.14.1: `receivedBytes===size && fileCompleteReceived && contiguous`) знімає цю гонку — завершення трактується лише за виконання всіх трьох умов.
- **Потоковий SHA-256 — механізм зафіксовано:** використовується **справжній інкрементальний хешер** над чанками у міру надходження (наприклад WASM/JS-хешер типу `hash-wasm` або інша бібліотека з update/finalize API). `crypto.subtle.digest` у WebCrypto є one-shot API й не дає переносимого проміжного стану, тому дозволений лише для малих/precomputed fallback-сценаріїв, де буферизація вкладається в ліміт пам'яті. **Заборонено** для проходження звірки великих файлів буферизувати весь файл у пам'ять — інакше великі файли (рівні §6.14.5) впруться в RAM. Отримувач хешує потоково паралельно із записом на диск.

#### 6.14.3 Timeout (детекція завислого трансферу)
- Якщо протягом `TRANSFER_STALL_MS` (дефолт 30 с) немає прогресу (у отримувача — нових чанків; у відправника — руху `bufferedAmount`/`file-progress`) → трансфер переходить у `failed` (`file-error{code:'STALL'}`), канал `file:<transferId>` закривається.

#### 6.14.4 Рівні resume (точна семантика)
`file-accept{offset=received}` продовжує передачу з байта `offset`. Межі відновлення — **три рівні**:
1. **DataChannel drop / ICE-restart, обидві вкладки живі** → resume **працює**. Стан трансферу (`transferId`, `received`, партіал на диску) живе в пам'яті/на диску вкладок; після відновлення каналу отримувач шле `file-accept{offset=received}`, відправник продовжує.
2. **Новий `RTCPeerConnection` у межах grace-періоду кімнати (§5.4)** → resume **працює** так само (кімната й peer-стан не зникли; `roomInstanceId` незмінний).
3. **Закриття будь-якої з двох вкладок** → resume **НЕможливий**. Стан трансферу тримається лише в пам'яті вкладок; його втрата вимагає нового `file-offer` (новий `transferId`, передача з нуля).
> **Прямо:** оскільки сервер **нічого не зберігає** (§1.1), відновлення після закриття обох вкладок **принципово неможливе** — це свідомий наслідок моделі приватності, а не дефект (див. §9.3).

#### 6.14.5 Політика пам'яті та ліміти
- **Отримувач пише потоково на диск**, а не буферизує файл повністю в RAM:
  - основний шлях — **File System Access API** (`showSaveFilePicker` → `FileSystemWritableFileStream`), чанки пишуться одразу, партіал переживає reconnect (рівні 1–2 вище);
  - **fallback** (Firefox, iOS Safari, де API немає) — накопичення в Blob-частинах у пам'яті з **нижчим** лімітом.
- **`MAX_FILE_BYTES` залежить від можливостей браузера:** повний ліміт (напр. 2 GiB) — лише коли доступний File System Access; для fallback-буферизації — знижений (напр. 256–512 MiB). Перевищення → `file-reject`/`file-error{code:'TOO_LARGE'}`.
- Одночасних активних трансферів ≤ `MAX_CONCURRENT_TRANSFERS` (дефолт 3); `chunkSize` — за замовчуванням 16 KiB.

## 7. Наскрізне шифрування (E2E)

- Кореневий link-ключ `K_link` (256 біт) генерується на клієнті-створювачі кімнати за допомогою **WebCrypto** і передається іншим учасникам **тільки через fragment лінки** (`#k=...`), який ніколи не надсилається на сервер.
- **Підключі виводяться через HKDF** (домен-сепарація, §7.1): `K_group` — для group AES-GCM, `K_auth` — для HMAC приватного сигналінгу. `K_link` **не** використовується напряму як ключ шифрування.
- Групові повідомлення шифруються AES-GCM (`K_group`, унікальний IV на кожне повідомлення) перед `relay`; метадані (`msgId/senderId/seq/ts`) входять в **AAD** (див. §6).

### 7.1 Автентичність приватного сигналінгу (захист від MITM)
Сервер контролює SDP і теоретично може підмінити DTLS-fingerprint, ставши посередником. Оскільки ключ із fragment серверу невідомий, ми прив'язуємо P2P-з'єднання до цього ключа.

**Окремий ключ автентифікації (не переюзаємо `#k`).** Link-ключ `K_link` не використовується напряму і для AES-GCM, і для HMAC. Виводимо призначені підключі:
```
K_auth  = HKDF-SHA256(ikm=K_link, salt=roomId, info="private-video-chat/webrtc-auth/v1", len=32)
K_group = HKDF-SHA256(ikm=K_link, salt=roomId, info="private-video-chat/group-aead/v1",  len=32)   // для §7 group AES-GCM
```

**Канонічний transcript — детерміноване кодування (без неоднозначності).**
```
authTranscript = CBOR(canonical, {
  1: "private-video-chat/webrtc-auth/v1",   // domain separation
  2: roomId,                                  // bstr/tstr
  3: roomInstanceId,                          // прив'язка до інстансу (§5.4)
  4: sort([nonceA, nonceB]),                  // ОБИДВА sessionNonce, відсортовані як байти
  5: sort([fpsA, fpsB])                       // нормалізовані fingerprint-набори обох сторін
})
```
- **Кодування — канонічний CBOR** (RFC 8949 core deterministic: впорядковані ключі, найкоротші int, визначені довжини). Це усуває проблему роздільника `:` всередині fingerprint (hex теж містить `:`) — байтові рядки length-prefixed, розділювачів немає. (Допустима альтернатива — length-prefixed binary або детермінований JSON з UTF-8; обрати **одну** й зафіксувати в `crypto.ts`.)
- **Обидва nonce** (`nonceA`, `nonceB`) з `PeerInfo` кожної сторони, **відсортовані** як байтові рядки → однаковий transcript незалежно від ролі. (Раніше було одне поле `sessionNonce` — виправлено.)
- `fpsX` — **усі** рядки `a=fingerprint` сторони X, нормалізовані (верхній регістр hex, розділювач `:`, з префіксом алгоритму), відсортовані між собою; потім два набори сортуються між собою.
- `roomInstanceId` **входить** у transcript.

**Обмін і перевірка:**
- Кожна сторона обчислює `mac = HMAC-SHA256(K_auth, authTranscript)`.
- Обмін `mac` через `auth`, звірка **constant-time**. Розбіжність = ймовірний MITM → розрив із попередженням.
- **Safety code** = `truncate(HKDF-SHA256(K_auth, authTranscript, "safety-code"))` → короткий рядок (6–8 груп). Містить реальні `fps` обох сторін, тож MITM зі своїм fingerprint **змінює код**.
- **КРИТИЧНИЙ ІНВАРІАНТ:** застосунок **не надсилає й не приймає жодних даних** (текст, файли, media tracks, DataChannel `chat`/`file:`) **до успішної взаємної перевірки `auth`**. Порядок: ICE/DTLS connected → обмін+звірка `mac` → лише потім `dc.send`/`addTrack`-потоки стають активними. До auth DataChannel може бути **відкритий**, але прикладні кадри в нього не йдуть.
- Примітка: окреме «додаткове AES поверх DataChannel» **не застосовується** — дублює DTLS; правильний захист — автентифікація fingerprint вище.

### 7.2 Автентичність відправника у групі
Спільний AES-ключ дає цілісність проти сторонніх, але **не** криптографічну автентичність між учасниками: будь-який власник ключа може вписати чужий `senderId` в AAD.

**Дворівнева модель у v1 (рішення зафіксовано — базовий рівень + зарезервований крипторівень):**
- **Рівень 1 (v1, реалізується): атрибуція за `participantId`.** Довірена ідентичність відправника — штамп `from: participantId` від сервера (§6.1, §6.4), а не `senderId` з AAD. За **чесного сервера** це дає практичний захист від інсайдера: учасник не може змусити сервер поставити чужий `participantId`. UI прив'язує ім'я/колір до `participantId`.
  - **Залишковий ризик:** *зловмисний* сервер може збрехати про `from`. Проти нього рівень 1 не захищає.
- **Рівень 2 (зарезервовано, §7.2(B)): per-sender Ed25519-підписи.** Поля `senderPubKey` (в `join`) і `sig` (в конверті) **вже присутні в протоколі v1** (§6.2), але у v1 не перевіряються. Увімкнення: кожен учасник публікує Ed25519-pubkey при join, підписує `(iv||ciphertext||aad)`; захист від підміни pubkey сервером — TOFU + **safety number** для звірки. Це закриває й зловмисний сервер.
> **Статус:** v1 постачається з **Рівнем 1**. Оскільки `senderPubKey`/`sig` уже в протоколі, перехід на Рівень 2 **не є breaking-зміною** — не потрібно міняти конверт після M1.

### 7.3 Forward secrecy та обмеження моделі довіри

**FS не пов'язана з lifecycle кімнати.** Знищення кімнати в пам'яті сервера — це серверна гігієна (відсутність персистенції), а **не** криптографічна властивість: лінка з `#k` багаторазова, тож повторний вхід за нею перевикористовує той самий ключ. Тому попереднє формулювання «forward secrecy через одноразовість кімнат» **хибне й прибрано**. FS у двох режимах різна:

- **Приватний (P2P): forward secrecy Є — від DTLS/SRTP.** Вміст (DataChannel, аудіо/відео) шифрується DTLS-ключами на **ефемерному ECDHE**, які WebRTC генерує й відкидає щосесії. Ключ із `#k` там вміст **не шифрує** — він лише автентифікує fingerprint (§7.1). Тому навіть витік лінки пізніше **не** розкриває записаний P2P-трафік минулих сесій. FS забезпечує WebRTC, а не lifecycle кімнати.
- **Груповий (relay): forward secrecy НЕМАЄ.** Контент шифрується **статичним** AES-ключем із fragment. Витік лінки → увесь записаний груповий шифротекст розшифровується. Ефемерність тут **лише серверна** (сервер нічого не зберігає), не криптографічна.
- **Немає re-keying і немає post-compromise security в межах кімнати:** ключ не ротується при зміні складу; **«кікнути» учасника неможливо** — будь-хто з лінкою може повернутись і читати трафік. Прямий наслідок статичного ключа в fragment (accepted, §9.3).
- **Реальна FS у групі** потребувала б ефемерного узгодження ключа (`#k` як кореневий/авторизаційний секрет + per-session DH); це окреме відкрите питання (§13), поза межами v1.

**Навіщо ключ окремо від room ID.** Room ID (шлях URL) сервер **зобов'язаний** знати для маршрутизації; ключ (`#k`, fragment) браузер серверу **ніколи не надсилає**. Доступ до кімнати гейтить **недовірений** сервер, а конфіденційність тримає секрет, якого сервер не має. Похідний ключ (`key=f(roomID)`) не годиться — сервер знає `roomID`, отже знав би й ключ. Тому room ID і ключ — **незалежні секрети** в різних частинах URL: room ID захищає від сторонніх, ключ — від сервера.

- **Доставка коду залишається довіреною**: сервер, що віддає JS/HTML, технічно може віддати шкідливий бандл і вивести ключ із fragment. Це фундаментальне обмеження браузерної E2E. Пом'якшення (конкретно):
  1. **окремий origin** для статики (ізоляція від API/сигналінгу);
  2. **строгий CSP** (`default-src 'self'`, заборона inline/eval, `connect-src` лише на власний WSS/TURN);
  3. **SRI** на всі бандли + (бажано) **підпис бандла** та **відтворювані білди**;
  4. чесно фіксуємо: єдиний *справжній* фікс — розширення браузера / нативний клієнт (поза v1). **Залишковий ризик.**

## 8. Конфігурація WebRTC / ICE

- `iceServers` (**залежить від середовища**):
  - **Локальна розробка / тестування:** дозволено публічний STUN `stun:stun.l.google.com:19302` — для зручності, без розгортання coturn.
  - **Продакшн:** **тільки власний STUN + TURN на coturn**. Публічний Google STUN у проді **не використовуємо** — він зливав би IP усіх учасників третій стороні (Google), що суперечить меті приватності (§9.1). Список `iceServers` формується бекендом залежно від `NODE_ENV`.
  - власний **TURN (coturn)** з обмеженими у часі обліковими даними (short-term credentials / REST API secret) — видаються сервером при join, не є постійними.
- `iceTransportPolicy`: `all` (relay використовується лише за потреби).
- coturn розгортається окремо (див. `infra/coturn/`), TLS для `turns:`.
- Ротація TURN-креденшелів: TTL коротких облікових даних (напр. 5–10 хв), генеруються на бекенді за спільним секретом.

### 8.1 Гігієна fragment-ключа
- Ключ `#k=` зчитується з `location.hash` **одразу при завантаженні**, після чого fragment **стирається з URL** через `history.replaceState` — щоб ключ не лишався в адресному рядку, історії вкладки чи не потрапив у скріншот.
- QR-код (FR-3) генерується **тільки на клієнті** (жодних сторонніх QR-сервісів — інакше ключ витік би на чужий сервер).
- Fragment ніколи не потрапляє в `Referer` (fragment за специфікацією не надсилається; додатково — `Referrer-Policy: no-referrer`).

## 9. Threat model (модель загроз)

### 9.1 Довірені та недовірені сторони

| Сторона | Рівень довіри | Що бачить / може |
|---------|---------------|-------------------|
| Учасник із лінкою | Довірена (в межах кімнати) | Весь відкритий вміст кімнати; володіє ключем з fragment |
| **Доставка коду** (сервер віддає JS/HTML) | **Довірена (вимушено)** | Може віддати шкідливий бандл і вивести ключ — E2E тримається лише на чесності доставки |
| Application/WS server (рантайм) | Недовірена щодо вмісту; **довірена лише щодо чесної маршрутизації** (`participantId`/`from`) | Шифротекст, сигнальні повідомлення; **не** бачить plaintext чи ключ; може спробувати MITM/replay/reorder; може збрехати про `from` (закривається §7.2 рівнем B) |
| STUN | Недовірена | IP/порти учасників. **Прод — власний STUN; публічний Google STUN лише для dev** (§8) |
| TURN (coturn) | Недовірена щодо вмісту | IP + релейний DTLS/SRTP-трафік (зашифрований), обсяг і тривалість сесії |
| Мережевий пасивний спостерігач | Недовірена | Лише TLS-трафік (wss/https), метадані з'єднань |

### 9.2 Загрози в межах захисту (in-scope) і контрзаходи
- **Пасивне читання сервером** → E2E AES-GCM; ключ лише у fragment (§7).
- **MITM приватного з'єднання недовіреним signaling-сервером (підміна SDP/fingerprint)** → HMAC(`K_auth`)-автентифікація транскрипту + safety code (§7.1). **Mitigated.** Safety code доводить «обидві сторони володіють `K_link`», а не «це саме той, кому я надіслав лінку» (див. §9.3).
- **Мережевий MITM без link-ключа** → той самий механізм (§7.1); атакуючий без `K_link` не відтворить `mac`. **Mitigated.**
- **Replay / перестановка групових повідомлень** → `seq`/`msgId` в AAD, дедуплікація (§6.4, §6.9).
- **Підміна метаданих повідомлення сервером** → поля в AAD, автентифіковані тегом GCM (§6.2).
- **Несумісність/downgrade протоколу** → обов'язкове `protocolVersion`, відхилення невідомих версій (§6.7).
- **Підробка `senderId` інсайдером у групі** → довірена атрибуція за штампом `from: participantId` (§6.1, §7.2 рівень 1); повний захист (проти зловмисного сервера) — рівень 2 (§7.2 B).
- **Slot-squatting раніше за легітимний reconnect** → `resumeToken` (§6.1): сторонній власник room ID **не може** зайняти чужий `participantId`-слот під час grace без токена. **Mitigated** (початкове зайняття слота — див. §9.3, це availability, не крипто-impersonation).
- **Зловживання ресурсами / fanout-амплікація** (група ретранслює 1→(n−1); Sybil) → rate-limit, cap розміру, `MAX_GROUP_PARTICIPANTS`, fanout-бюджет, ліміти per-IP, heartbeat (§10.5).

### 9.3 Залишкові (accepted) ризики та поза межами v1
- **Шкідливий JS від сервера / XSS** — браузерна E2E не може це закрити; пом'якшення: строгий CSP, SRI, відтворювані білди, окремий origin (§7.3). **Залишковий ризик.**
- **Автентичність відправника у групі проти зловмисного сервера** — v1 постачається з рівнем 1 (`participantId`), що захищає лише від інсайдера за чесного сервера. Криптографічний рівень 2 (§7.2 B) — поза межами v1 (поля вже зарезервовано).
- **Відсутність re-keying / «кіку» в межах кімнати** — учасник, що вийшов, може повернутись за лінкою й читати трафік; ключ не ротується (§7.3). **Accepted.**
- **Bearer-link ризик: перехоплення або пересилання повної лінки третій особі.** Модель довіри — «хто має лінку, той у кімнаті». Той, хто отримав повну лінку (з `#k`), володіє `K_link` → створить коректний `mac` і коректний safety code. Тому safety code **не** підтверджує «це саме та людина, якій я надіслав посилання» — лише «обидві сторони мають секрет». Захист від MITM без ключа (§9.2) при цьому лишається. **Accepted bearer-link risk** — поза криптографічним захистом v1 (пом'якшення: канал доставки лінки).
- **Початкове зайняття слота приватної кімнати (initial squatting)** — до першого reconnect `resumeToken` ще не видано, тож найперший, хто відкрив лінку, стає peer #1 (за задумом, FR-21). Це **availability/DoS**, а не крипто-impersonation: підміну другого peer-а під час сесії ловить safety code (§7.1) за звірки користувачами; повторне зайняття після виходу закриває tombstone (§5.4). **Частково accepted.**
- **Відсутність forward secrecy у груповому чаті** — контент шифрується статичним ключем із fragment; витік лінки розкриває записаний груповий трафік (§7.3). Реальна FS потребує ефемерного DH (§13). У private FS є (DTLS). **Accepted для групи у v1.**
- **Метадані на app-server / STUN / TURN** — IP, тайминги, обсяги, факт і тривалість дзвінка приховати не можемо (без onion-маршрутизації); задокументовано як відомий leak.
- **Компрометація пристрою учасника, скріншоти, пересилання лінки далі** — поза моделлю.
- **Аналіз трафіку (розмір/тайминг повідомлень)** — не мітигуємо у v1 (padding/cover traffic — на майбутнє).
- **Втрата повідомлень під час реконекту WS у групі** — за дизайном (без історії/буферизації) повідомлення, надіслані під час дискнекту, до клієнта не доходять. **Accepted UX.**
- **Неможливість resume файлового трансферу після закриття вкладки** — стан трансферу тримається лише в пам'яті/на диску вкладок; закриття будь-якої з двох вкладок (або вихід за межі grace-періоду кімнати) втрачає його безповоротно → потрібен новий `file-offer` з нуля. Прямий наслідок «сервер нічого не зберігає» (§1.1, §6.14.4). **Accepted UX.**

### 10.5 Ліміти та бюджети (v1, стартові дефолти)
Усі значення — **конфігуровані** (`packages/shared/src/constants.ts` + env). Дефолти нижче — **гіпотези** для одного процесу-інстансу (§10.6), що **затверджуються load/TURN-тестами** (§14), а не остаточні числа. Кожен бюджет має явний **scope** (одиниця обліку) та **вікно/burst**.

| Бюджет | Константа | Дефолт | Scope / одиниця обліку | Вікно / burst | При перевищенні |
|--------|-----------|--------|------------------------|---------------|-----------------|
| Учасники групової кімнати | `MAX_GROUP_PARTICIPANTS` | 50 | per-room (одночасні) | — | `ROOM_FULL` новому join |
| Учасники приватної кімнати | (фіксовано) | 2 | per-room | — | `ROOM_FULL` (FR-13) |
| Активні WS на процес | `MAX_CONNECTIONS` | 10 000 | per-process | — | відмова upgrade → `CAPACITY` (1013) |
| Активні кімнати на процес | `MAX_ROOMS` | 5 000 | per-process | — | `error{CAPACITY}` (не `INTERNAL`) |
| Кімнати на IP | `MAX_ROOMS_PER_IP` | 20 | per-IP (одночасні) | миттєвий | `RATE_LIMITED` на join/create |
| Створення кімнат | `CREATE_ROOMS_PER_IP` | 10 / хв | per-IP | ковзне 60 с, burst 5 | `RATE_LIMITED` |
| Join | `JOINS_PER_IP` | 30 / хв | per-IP | ковзне 60 с, burst 10 | `RATE_LIMITED` |
| WS-конекти на IP | `CONNS_PER_IP` | 50 | per-IP (одночасні) | миттєвий | відмова upgrade (429) |
| Relay msgs | `RELAY_MSGS_PER_SEC` | 20 / с | **per-participant (ingress від відправника)** | токен-бакет 1 с, burst 40 | `RATE_LIMITED`; після N поспіль — `fatal` |
| Relay bytes | `RELAY_BYTES_PER_SEC` | 256 KiB / с | **per-participant (ingress)** | токен-бакет 1 с | `RATE_LIMITED` |
| Розмір WS-фрейму / ciphertext | §6.6 | 64 KiB / 16 KiB | per-message | — | `MSG_TOO_LARGE` |
| WS backpressure | `MAX_BACKPRESSURE_BYTES` | 1 MiB | **per-socket (egress buffer)** | миттєвий | `BACKPRESSURE`, close 4413 |
| Fanout egress | `FANOUT_BYTES_PER_SEC` | 4 MiB / с | **per-room сумарний egress** (усі копії всім отримувачам) | токен-бакет 1 с | throttle кімнати; повільні отримувачі — дроп по backpressure |
| TURN: смуга на allocation | coturn `max-bps` | 2 Mbps | per-allocation | — | coturn обмежує потік |
| TURN: квота користувач/тотал | coturn `user-quota`/`total-quota` | за деплоєм | per-user / per-server | — | coturn відхиляє нові allocation |
| TURN: allocation на IP | (бекенд) | 10 | per-IP (одночасні) | — | відмова видачі креденшелів |

- **Уточнення scope, що були неоднозначні:**
  - `FANOUT_BYTES_PER_SEC` — **сумарний egress кімнати** (сума всіх ретрансльованих копій), а не per-конект; захищає від амплікації `1 msg → (n−1)` доставок. Окремо `RELAY_*` рахуються на **вході** від відправника (ingress), `MAX_BACKPRESSURE_BYTES` — на **виході** конкретного сокета.
  - `MAX_ROOMS`/`MAX_CONNECTIONS` → `CAPACITY` (1013 «спробуйте пізніше»), **не** `INTERNAL`: вичерпання ємності не є збоєм.
- **Джерело client IP:** за реверс-проксі/LB IP береться з `X-Forwarded-For`/`Forwarded` **лише** якщо запит прийшов від **довіреного проксі** (allow-list `TRUSTED_PROXIES`); інакше — peer-адреса сокета (інакше per-IP-ліміти тривіально обходяться).
- **IPv6-нормалізація:** per-IP-облік для IPv6 ведеться на рівні **/64-префікса** (а не повної адреси), бо провайдери видають /64+ одному користувачу.
- **Спільний NAT (CGNAT/корпоративний):** per-IP-ліміти свідомо **грубі** — багато користувачів за одним IP ділять бюджет. Дефолти обрані з запасом; за потреби — окремий вищий ліміт для відомих NAT-діапазонів (поза v1).
- **TURN 2 Mbps** прив'язано до **цільового відео**: 720p@30fps VP8/VP9 ≈ 1–1.5 Mbps + аудіо ≈ 40 kbps + запас; для 1080p підняти до ~3 Mbps. Значення затверджується TURN-тестом (§14) під реальний bitrate/роздільність.
- Rate-limit — токен-бакет; лічильники **в пам'яті процесу** (ефемерні, як кімнати).

### 10.6 Модель розгортання та масштабування
- **v1 — один stateful інстанс.** Кімнати, `participantId`↔`connId`-мапінг, `resumeToken`-и, `seq`-стан, rate-лічильники й tombstone-и живуть **у пам'яті одного процесу**. Це свідомий вибір для мінімалізму v1.
- **Уточнення до принципу приватності:** «сервер нічого не зберігає» (§1.1) стосується **персистентного вмісту** (немає БД, логів переписки/файлів). Ефемерний **рантайм-стан** (реєстр кімнат у RAM) існує — і завжди існував; він зникає з процесом і не є персистенцією.
- **Горизонтальне масштабування — поза v1** і **змінює модель стану**. Кімната має жити в одному місці, тому потрібно одне з:
  1. **sticky routing** — усі WS однієї кімнати спрямовуються на той самий інстанс (консистентне хешування `roomId` на LB/edge); проста, але нерівномірне навантаження;
  2. **зовнішній ephemeral coordination layer** (напр. Redis pub/sub для fanout + presence) — рівномірніше, але додає **зовнішній компонент**, який *бачить метадані кімнат* (room IDs, хто-де), тобто **розширює поверхню довіри** й вимагає перегляду §9 (E2E-вміст лишається закритим, але метадані течуть у ще одну систему).
- **Наслідок для ТЗ:** будь-яке з рішень масштабування — це **окреме рішення з оновленням threat model**; у v1 фіксуємо single-instance і документуємо межу зростання (обмежена бюджетами §10.5 на один процес).

## 13. Відкриті питання (на майбутнє)

- **Forward secrecy у груповому чаті через ефемерний DH.** У v1 груповий контент шифрується статичним ключем із fragment → FS немає (§7.3). Справжня FS потребує, щоб `#k` став **кореневим/авторизаційним** секретом, а контент шифрувався **per-session** ключем із ефемерного узгодження (X25519). Складність: у групі з N учасників і недовіреним реле це вимагає групового key-agreement (sender keys / MLS-подібний під-протокол), що суперечить меті мінімалізму v1. Рішення: **відкладено**, зафіксувати як свідомий trade-off.
- **Криптографічна автентичність відправника у групі (§7.2, рівень 2)** — Ed25519-підписи; поля `senderPubKey`/`sig` уже зарезервовано.
- Групові аудіо/відеодзвінки (потрібен SFU) — окремий проєкт/етап.
- PWA / офлайн-оболонка.
- Опційний парольний захист лінки поверх ключа з fragment.
- i18n (наразі — українська за замовчуванням).

## 14. Критерії приймання (Definition of Done)

- Дві робочі лінки (group / private) створюються зі стартової сторінки.
- Групова лінка: обмін повідомленнями між ≥ 3 учасниками; сервер не зберігає нічого; повідомлення E2E-зашифровані у транспорті.
- Приватна лінка: між двома учасниками працюють повідомлення, передача файлу, аудіо- та відеодзвінок повністю через WebRTC.
- Встановлення з'єднання успішне за симетричним NAT (через TURN).
- Приватне з'єднання автентифікує DTLS-fingerprint ключем `K_auth` (канонічний CBOR-transcript із **обома** `sessionNonce` + `roomInstanceId`, §7.1); safety code, похідний від реальних fingerprint-ів, збігається в обох учасників; підміна SDP сервером виявляється й розриває з'єднання; **жодні дані не йдуть до успішного auth**.
- Групові повідомлення атрибутуються за штампом `from: participantId`; відхиляються при replay/перестановці (`seq` per-`participantId`, зберігається через реконект) та дедуплікуються за `msgId`.
- Прод-профіль ICE не містить сторонніх STUN/TURN; fragment стирається з URL після зчитування; access-логи не персистять метадані.
- Threat model (§9) актуальний; §7.2 зафіксовано (v1 = рівень 1 `participantId`, поля `senderPubKey`/`sig` зарезервовано під рівень 2).
- FS-модель зафіксована й несуперечлива: private має FS через DTLS, group — ні (§7.3); термін «одноразова кімната» в документі відсутній.
- Короткий обрив мережі (реконект WS / ICE-restart у межах grace) **не** руйнує кімнату — відновлюється той самий `participantId`/`roomInstanceId` за `resumeToken` (§5.4, §6.1).

### 14.1 Тестова матриця (метод перевірки кожного критерію)
DoD вважається виконаним лише за проходження матриці:

| Область | Тест / метод | Пройдено, якщо |
|---------|--------------|----------------|
| Сумісність | пінована матриця **browser × version × OS** (§10.3): Chrome N/N−1, Firefox ESR+stable, Safari 2 мажорні, Edge stable; desktop+mobile | усі клітинки: створення кімнати + текст + P2P |
| Транспорт NAT | **direct P2P** і **forced TURN** (`iceTransportPolicy:'relay'`) окремо; симетричний NAT емульовано | з'єднання встановлюється в обох режимах |
| Продуктивність | Lighthouse-профіль (NFR-6, p75) + load-тест сигналінгу p95/p99 (NFR-7) під 50 кімнат | у межах порогів |
| Reconnect | обрив **до** та **після** `ROOM_GRACE_MS`; reclaim за `resumeToken` | до grace — той самий `participantId`; після — новий, стан скинуто |
| Slot reclaim | конкурентна спроба зайняти слот під час grace без токена | відхилено (`BAD_RESUME_TOKEN`/`ROOM_FULL`), легітимний reconnect успішний |
| Крипто | tamper-AAD → провал GCM; `mac`-mismatch → розрив; **auth-before-data** (жодного кадру до auth) | усі виконуються |
| Файли | resume з offset; **channel reordering** (`file-complete` раніше чанків, §6.14.2); checksum mismatch; memory-ceiling на великому файлі (потоковий запис, RAM не росте) | коректна поведінка в кожному |
| Навантаження | load-тест **50-member** групи: fanout, rate-limit, backpressure | без OOM/деградації понад бюджети §10.5 |
| «Сервер не зберігає» | **метод перевірки:** grep процес-логів на відсутність ciphertext/plaintext; heap-dump/інспекція після `leave` — немає лишків повідомлень; tombstone очищується за `ROOM_TOMBSTONE_MS` | усі три підтверджені |
| Запуск | **точні команди:** `pnpm install && pnpm build && pnpm start` (dev: `pnpm dev`) — задокументовані в README і виконуються з чистого клону | працює без ручних кроків |

- Монорепо збирається та запускається задокументованими командами (`pnpm install`, `pnpm build`, `pnpm start` / `pnpm dev`).
